    Python package

    The KITE package for pre-processing is split up in various subclasses and contains several functions:

    • class kite.StructuralDisorder(lattice, concentration=0, position=None) - Add disorder to the lattice.
    • class kite.Disorder(lattice) - Add Guassian disorder to the lattice.
    • class kite.Modification(magnetic_field=None, flux=None) - Add a magnetic field to the lattice.
    • class kite.Configuration([...]) - Define the basic parameters used in the calculation.
    • class kite.Calculation(configuration=None) - Describe the required target functions.
    • function make_pybinding_model
    • function estimate_bounds
    • function config_system
    • warning LoudDeprecationWarning

    functionldos_map(energy_, sigma_, vectors_, coef="gaussian")

    Calculate a full real-space map of the local density of states at one target energy.

    KITE evaluates this at every lattice site simultaneously with a stochastic (random-phase-vector) Chebyshev/KPM estimator combined with a Gaussian- or window-broadened energy filter \(f(H)\). Up to the overall normalization, the stored quantity is \(\mathbb{E}[\text{map}_r] = \text{factor}\cdot\langle r|f(H)^2|r\rangle\) — a filtered-random-state local-intensity estimator, not simply \(\langle r|f(H)|r\rangle\). This is a standard stochastic-trace technique; see the review by Weiße, Wellein, Alvermann, and Fehske.1

    Parameters
    Parameter Description
    energy_:float Target energy where the map is evaluated.
    sigma_:float Width of the Gaussian (or window) that approximates the Dirac delta.
    vectors_:int Number of independent random-phase vectors to average over (analogous to num_random for dos() — not a number of k-points or spatial points).
    coef:str Choice of energy-filter kernel, either "gaussian" or "window".

    Disorder is not ensemble-averaged

    Unlike dos() and arpes(), ldos_map() has no num_disorder parameter: a single disorder realisation is generated per call. This is a deliberate design choice appropriate for visualizing the real-space structure of one disorder realization (e.g. impurity resonances), not an oversight — but the output of a single call should not be interpreted as a disorder-ensemble average.

    is_complex=True is mandatory, even for real-valued hoppings

    ldos_map() (and spectral_map()) are only compiled into KITEx for the complex Hamiltonian instantiation (if constexpr (is_tt<std::complex, T>::value) in Src/Simulation/SimulationLDoS.cpp). With is_complex=False, that branch is compiled away entirely — KITEx runs, prints "Calculating LDoS. Done.", and exits normally, but no /Calculation/ldos_map/Map dataset is ever written. This is a silent no-op, not an error: verified directly by running with is_complex=False and finding the expected dataset simply absent from the output file. Always set is_complex=True for these two methods, even if every hopping in your lattice is real.

    Example

    examples/piflux_ldos_map.py maps the LDOS at the Dirac-point energy (\(E=0\)) around a single vacancy in a \(\pi\)-flux square lattice — the LDOS vanishes exactly at the vacancy site and shows a decaying, oscillatory enhancement around it. See the [in-depth write-up][markov_maps_example] for the full explanation and plots. This is the reference method for stochastic real-space/momentum-space maps with a rigorous, Markov-inequality-bounded per-site sampling error.3

    functionspectral_map(energy_, sigma_, vectors_, coef="gaussian")

    Calculate a full momentum-space (k-resolved) map of the spectral function at one target energy — the k-resolved analogue of ldos_map(). It uses the same random-phase/Chebyshev-filter machinery, but wraps the filtering loop in a unitary FFT that transforms between the real-space (site) and momentum-space (k) bases — a proper Bloch transform that includes intra-cell sublattice phases, not a naive plane DFT — before and after applying \(H\): \(\mathbb{E}[\text{map}_k] = \text{factor}\cdot\langle k|f(H)^2|k\rangle\).

    Parameters
    Parameter Description
    energy_:float Target energy where the map is evaluated.
    sigma_:float Width of the Gaussian (or window) that approximates the Dirac delta.
    vectors_:int Number of independent random-phase vectors to average over.
    coef:str Choice of energy-filter kernel, either "gaussian" or "window".

    Relation to arpes()

    spectral_map() is conceptually the "all-k-at-once" stochastic sibling of arpes(), which instead evaluates exact Chebyshev moments at a handful of user-chosen k-points via a genuine plane-wave state (no FFT, no randomness). The FFT in spectral_map() only does non-trivial work because real-space disorder breaks translation symmetry inside the sandwiched Chebyshev recursion — for a clean/periodic system it would trivially return the same map at every k-point regardless of averaging.

    Disorder and boundary conditions are not ensemble-averaged consistently

    As with ldos_map(), disorder is frozen per call (no num_disorder). In addition, if you configure random or twisted boundary conditions (see kite.Configuration), each realization samples a different, randomly-twisted boundary condition while the FFT's k-grid is fixed from the untwisted lattice size — averaging realizations under that combination is not accounted for by the normalization and may not do what you expect. Fixed (non-twisted) boundaries are recommended when using spectral_map() unless you specifically understand this interaction.

    The FFT runs over every axis, regardless of its declared boundary condition

    Src/FFT/FFT.cpp performs the momentum-space transform over all axes unconditionally, even an axis declared "open" (a real, hard-wall boundary where momentum isn't actually conserved). Verified directly in examples/weyl_spectral_map.py: with boundaries=["open", "random", "random"], the spectral weight along the two periodic/twisted axes is sharply peaked, while along the open axis it is smeared and nearly flat — a real, physically-expected consequence of transforming an axis with no translational symmetry, not a bug, but easy to misread as a broken/noisy result if you don't expect it.

    Example

    examples/weyl_spectral_map.py computes \(A(\mathbf{k}, E{=}0.7)\) for a 3D Weyl semimetal and shows two clean ring contours — constant-energy cuts through the two Weyl nodes' linearly-dispersing cones. See the [in-depth write-up][markov_maps_example] for the full explanation and plots.

    See the base KITE method paper2 for the general LDOS/ARPES-of-disordered-materials methodology demonstrated by this family of target functions; the specific real-space↔k-space basis-change construction used internally by spectral_map() does not have a separately located dedicated reference.

    functiongaussian_wave_packet(num_points, num_moments, timestep, k_vector, spinor, width, mean_value, num_disorder=1, operators=None, probing_point=0)

    Calculate the time evolution function of a wave packet.

    Parameters
    Parameter Description
    num_points:int Number of time points for the time evolution.
    num_moments:int Number of polynomials in the Chebyshev expansion.
    timestep:float Timestep for calculation of time evolution.
    k_vector:array_like Different wave vectors, components corresponding to vectors b0 and b1.
    spinor:array_like Spinors for each of the k vectors.
    width:float Width of the gaussian.
    mean_value:tuple(float, float) Mean value of the gaussian envelope.
    num_disorder:int Number of different disorder realisations.
    operators:list[str] Labels of operators registered via add_orbital_coupling(), whose expectation value is tracked at every timestep (e.g. ['l0', 'l1', 'l2'] for a 3-component spin, orbital-angular-momentum, or quadrupole operator set).
    probing_point:int or array_like Forward probing point, defined with x, y coordinate were the wavepacket will be checked at different timesteps.
    Tracked operators

    gaussian_wave_packet() has no built-in notion of spin: any operator, including spin, is tracked through the same add_orbital_coupling() mechanism used by custom_one()/custom_two() — including its on-site-only restriction (see the warning under add_orbital_coupling()): a tracked operator must be expressible as a single matrix acting on one site's orbitals, identical at every site. Spin, orbital angular momentum, and orbital quadrupole operators satisfy this; an operator that couples different sites cannot be tracked this way. Each requested label's expectation value \(\langle\psi(t)|\hat O|\psi(t)\rangle\) is written to /Calculation/gaussian_wave_packet/<label> in the output file. Unlike custom_one()'s stream_ tokens, these labels are read back by an explicit ordered list, not the single-digit "l0"–"l9" scheme described in the warning above — so this path does not share that 10-operator limit.

    functionlocalized_wave_packet(time, num_measures, initial_pos, num_moments=0, width=-1., energy_window=[0.,0.], initial_wavevector=None, probes=None, sample_start=-1, sample_L=-1)

    Calculate the time evolution of a localized (point-like) or Gaussian-envelope wave packet, with optional spectral filtering. This is distinct from gaussian_wave_packet(): the initial state is dispatched between a point-like or a Gaussian-envelope wave packet depending on whether width > 0, it can be band/energy-filtered to energy_window before propagation, and it reports richer observables (spatial spread statistics, return probability, propagator amplitude, spectral moments, and transmission weights at chosen probe positions) than the plain spin expectation values returned by gaussian_wave_packet().

    Parameters
    Parameter Description
    time:float Total dimensionless time over which the wave packet is evolved. Together with num_measures, this sets the time step between measurements.
    num_measures:int Number of times the evolved state is probed.
    initial_pos:array_like Initial position of the localized wave packet in lattice coordinates, [n_x, n_y[, n_z], n_orbital].
    num_moments:int Number of Chebyshev moments to calculate for the spectral function of the propagated state.
    width:float Width of the Gaussian envelope in real space. If width <= 0 a point-like (not Gaussian-enveloped) wave packet is used.
    energy_window:array_like The localized packet is filtered to keep only eigenstates with energy inside this window.
    initial_wavevector:array_like Wavevector the Gaussian envelope is centered around, in reciprocal lattice coordinates.
    probes:array_like List of positions, in lattice coordinates, where the propagator/transmission weights are evaluated.
    sample_start:int Start (in the x-direction) of a disordered sample region embedded between clean leads, for transport/localization-length studies.
    sample_L:int Length of the disordered sample region described by sample_start.

    There is currently no example script in examples/ that uses localized_wave_packet(). See the C++ implementation (Src/Simulation/SimulationLocalizedWavePacket.cpp) and the parameter reference above for now.

    functionconductivity_dc(direction, num_points, num_moments, num_random, num_disorder=1, temperature=0)

    Calculate the DC conductivity for a given direction.

    Output units: e²/h

    The reconstructed conductivity (via KITE-tools --CondDC) is in units of \(e^2/h\) directly — not \(e^2/\hbar\), \(e^2/(\pi h)\), or any other common convention. Verified empirically (not just asserted): the Hall conductivity \(\sigma_{xy}\) of a Haldane Chern insulator (\(C=1\), examples/dos_dccond_haldane.py's own parameters, including its Anderson disorder) reconstructs to \(\sigma_{xy}\approx1.01\)–\(1.02\) at mid-gap — a topological plateau quantized at exactly \(C\cdot e^2/h=1\,e^2/h\), insensitive to broadening, disorder, or moment-count details, unlike a semiclassical Drude-regime check would be. The same convention applies to conductivity_optical() and the custom_two() family (all built from the same underlying Kubo-Bastin/Gamma2D machinery).

    Parameters
    Parameter Description
    direction:str Direction in \(xyz\)-coordinates along which the conductivity is calculated, supports "xx", "yy", "zz", "xy", "xz", "yx", "yz", "zx", "zy".
    num_points:int Number of energy point inside the spectrum at which the DOS will be calculated.
    num_moments:int Number of polynomials in the Chebyshev expansion.
    num_random:int Number of random vectors to use for the stochastic evaluation of trace.
    num_disorder:int Number of different disorder realisations.
    temperature:float Value of the temperature at which we calculate the response. If \(eV\) is used as unit for energy, then \(k_B\cdot T\) is also in \(eV\). To define the temperature in arbitraty units, specify the quantity \(K_B \cdot T\), which has units of energy.

    functionadd_orbital_index(label_, idx_)

    Register a name for an orbital index, for later reference inside a custom operator string (see custom_one() below). Must be called once per orbital before using add_orbital_coupling().

    Parameters
    Parameter Description
    label_:str Name used to refer to this orbital when calling add_orbital_coupling().
    idx_:int Index (0-based) assigned to this orbital inside the custom orbital-space operator matrices.

    functionadd_orbital_coupling(start_, last_, c_, label_)

    Set one matrix element of a named, custom orbital-space operator matrix \(M_{\texttt{label\_}}\), for use as an "l"-type token inside a custom operator string (see below). Internally this sets \(M_{\texttt{label\_}}[\text{idx}(\texttt{last\_}),\,\text{idx}(\texttt{start\_})] = \texttt{c\_}\), where idx(...) is the mapping registered with add_orbital_index(). Calling this repeatedly with the same label_ accumulates further matrix elements into the same operator matrix (which is otherwise initialized to zero).

    Parameters
    Parameter Description
    start_:str Name (registered via add_orbital_index()) of the orbital coupled from.
    last_:str Name of the orbital coupled to.
    c_:complex Value of the matrix element.
    label_:str Name of this custom orbital operator; must begin with "l" (raises ValueError otherwise).

    \"l\"-type operators are on-site only

    \(M_{\texttt{label\_}}\) is a single orbital-space matrix, applied identically and independently at every lattice site — it can only mix orbitals belonging to the same site, never couple different sites or unit cells. This is why it's a separate mechanism from the "v" (velocity) token: velocity is intrinsically an inter-site (hopping) operator, computed a completely different way (h.build_velocity()), not expressible as a per-site matrix. Spin, orbital angular momentum, and orbital quadrupole operators are all valid "l"-type operators because they are genuinely on-site quantities (built entirely from a single site's orbital manifold, e.g. the \(p_x,p_y,p_z\) orbitals of one atom) — but this mechanism cannot represent an operator that is diagonal in orbital index yet varies from site to site, nor one with any inter-site matrix element.

    The kite.custom.Vertex operator language

    A kite.custom.Vertex(num_moments, stream) object (import with from kite import custom) defines a custom operator \(J = \sum_i c_i O_i\) as a weighted sum of operator-string products, to be traced against Chebyshev polynomials of \(H\). num_moments is the number of Chebyshev moments computed for this vertex — note that in the shipped example scripts this argument is passed as a variable named pol_A/pol_B, which is not a polarization, it is a moment count. stream is a list of [coefficient, operator_string] pairs; operators can also be appended afterwards with vertex.add_operator(coef, operator_string).

    Each operator_string is a dot-separated chain of tokens, e.g. "vy.rx". The first character of each token selects the operator type:

    • "v" + component (e.g. "vy") — the velocity operator along that Cartesian direction ("x", "y", or "z").
    • "r" + component (e.g. "rx") — the real-space position operator along that direction.
    • "l" + digit (e.g. "l0") — a custom orbital-space matrix registered with add_orbital_index() / add_orbital_coupling().

    A dotted chain composes the tokens as an operator product applied to the state right-to-left as written, i.e. "vy.rx" applies "rx" to the state first and then "vy" — equivalent to the ordinary matrix product \(v_y r_x\) acting on a ket.

    Only single-digit orbital-operator indices are supported

    Because the C++ reader consumes exactly one character after "l", only labels "l0"–"l9" can be addressed inside an operator string — a hard limit of 10 distinct custom orbital operators per model. The token's digit selects the orbital operator positionally from the operators exported to the HDF5 file; this was not independently verified end-to-end against the HDF5 read-back order beyond the "l0", "l1", ... naming convention, so it is safest to register your custom orbital operators as "l0", "l1", "l2", … in the same order you intend to reference them.

    functioncustom_one(stream_, num_random_, num_disorder_)

    Calculate the generalized rank-one KPM trace Tr[Tn(H)·J] for an arbitrary custom operator \(J\) built from a kite.custom.Vertex (see the operator-language box above).

    Parameters
    Parameter Description
    stream_:kite.custom.Vertex Vertex object defining the operator \(J\) as a weighted sum of operator strings.
    num_random_:int Number of random vectors to use for the stochastic evaluation of the trace.
    num_disorder_:int Number of different disorder realisations.

    The \"v\" (velocity) building block and NumVelocities

    KITE's raw "vx"/"vy" DSL token is \([\hat H,\hat r]\) — missing the \(i/\hbar\) factor of the textbook Hermitian velocity operator. custom_one() counts how many "v"-type tokens appear in stream_ itself (every term must carry the same count, or a ValueError is raised) and writes it as NumVelocities in the output file. This determines whether Tr[Tn(H)·J] is physically real (even count) or purely imaginary (odd count) — both are genuine signal, not noise — and lets KITE-tools --CustomOne reconstruct the correct component automatically. You do not need to insert a compensating i into the vertex yourself; see examples/haldane_orbital_magnetization.py for a worked example with an odd (single) velocity operator.

    Worked example

    examples/haldane_orbital_magnetization.py computes the orbital magnetization of a Haldane-model Chern insulator via custom_one(), post-processed with KITE-tools --CustomOne.

    functioncustom_singleshot_two(stream_, num_random_, num_disorder_, gamma_, sigma_, energies_)

    The energy-resolved, single-shot sibling of custom_two(): evaluated directly by KITEx at fixed broadening and a chosen list of energies, analogous to how singleshot_conductivity_dc() relates to conductivity_dc() — no KITE-tools post-processing step is required.

    Parameters
    Parameter Description
    stream_:list(kite.custom.Vertex, kite.custom.Vertex) The two vertices [A, B] defining the operators \(A\) and \(B\).
    num_random_:int Number of random vectors to use for the stochastic evaluation of the trace.
    num_disorder_:int Number of different disorder realisations.
    gamma_:float Imaginary broadening (self-energy) parameter, as in singleshot_conductivity_dc()'s eta.
    sigma_:float Width of the Gaussian broadening applied to the reconstructed spectral quantity.
    energies_:array_like List of energies at which custom_singleshot_two() will be calculated.

    energies_/gamma_/sigma_ are not shifted by energy_shift

    Unlike singleshot_conductivity_dc(), which subtracts config.energy_shift from energy before exporting it, custom_singleshot_two()'s energies_/gamma_/sigma_ are only divided by config.energy_scale (this division happens on the KITEx/C++ side, not in Python) — energy_shift is never subtracted. If your spectrum_range is not symmetric about zero, energies_ is measured relative to the Hamiltonian's raw zero, not the center of your configured spectrum_range.

    No worked example yet

    There is currently no example script that uses custom_singleshot_two().

    kite.visualize¶

    Pure-Python (numpy/matplotlib/scipy) tools for inspecting a native kite.lattice.Lattice directly — unit cell, Brillouin zone, k-paths, and tight-binding band structure — with no dependency on KITEx/KITE-tools or pybinding. Meant as a fast sanity check of a lattice definition (wrong sign, wrong hopping, wrong lattice vector) before paying for an expensive KPM/Chebyshev run, not as a replacement for KITE's own stochastic machinery. Import as from kite importvisualize as viz.

    function viz.make_path(*points, step=0.1, point_labels=None)

    Build a piecewise-linear k-path through a sequence of high-symmetry points, with point density set by each segment's actual Cartesian reciprocal-space distance (not a fixed count per segment, and not fractional-coordinate distance). points must be given in Cartesian reciprocal-space coordinates — the same units lattice.reciprocal_vectors() returns — not fractional/reduced coordinates (a native kite.lattice.Lattice method; no separate API doc page exists for this class yet). Returns (k_points, tick_indices, tick_labels); tick_indices locates each input high-symmetry point's row in k_points, for tick placement on a band-structure plot's x-axis.

    function viz.plot_unit_cell(lattice, repeat=(1, 1), ax=None, sublattice_colors=None)

    Scatter each sublattice's position (stably colored/labeled by name) and draw a line for every hopping term to its actual neighbor (including across unit-cell boundaries), repeated over a small window of neighboring cells so boundary-crossing bonds are visible. 2D lattices only.

    function viz.plot_brillouin_zone(lattice, ax=None, k_path=None, length_unit="nm")

    Draw the 2D Brillouin zone (the Voronoi cell of the reciprocal lattice around \(\mathbf k=0\), via lattice.brillouin_zone()) plus the reciprocal lattice vectors. k_path optionally overlays a path — either raw points, or make_path(...)'s full 3-tuple return value directly (in which case high-symmetry points get marked and labeled). 2D lattices only.

    function viz.hamiltonian_k(lattice, k)

    Build the Bloch tight-binding Hamiltonian \(H(\mathbf k)\) directly from a lattice's own hoppings — one sublattice = one orbital (kite.lattice.Lattice only stores a scalar onsite/hopping value per sublattice pair; multi-orbital-per-sublattice matrices are a separate kite.custom concern for building observables, not this base Hamiltonian).

    Gauge convention (derived, not assumed)

    Matches KITE's own ARPES plane-wave state (the full atomic position, cell + sublattice offset, enters the phase — see Spectral Function). For a hopping family with relative_index=R, from_sub=a, to_sub=b, energy=t: $$ H_{ba}(\mathbf k)\mathrel{+}= t\;e^{-i\,\mathbf k\cdot(\mathbf R_{\text{cart}}+\mathbf d_b-\mathbf d_a)} $$ with a minus sign in the exponent — get this backwards and centrosymmetric bands (e.g. graphene) still look right while anything valley/Rashba/Weyl-asymmetric comes out mirror-flipped. Since add_one_hopping only ever stores one bond direction, this function builds the one-directional sum \(H_0(\mathbf k)\) and returns \(H_0(\mathbf k)+H_0(\mathbf k)^\dagger+\text{onsite}\), reproducing the missing reverse-direction terms automatically. The result is checked numerically Hermitian before being returned; a non-Hermitian result raises RuntimeError rather than being silently returned.

    Verified against several known models, not just graphene

    examples/band_structure_graphene.py checks the three textbook graphene energies directly: \(E(\Gamma)=\pm3t\), \(E(M)=\pm t\), \(E(K)=0\) (the Dirac point, exactly two-fold degenerate) — all three reproduced to floating-point precision. Also checked (not shipped as example scripts, but confirmed while building this feature): the Haldane model's gap (\(2\times3\sqrt3\,t_2\sin\phi\)) matches exactly at both inequivalent BZ corners; kite.repository.group6_tmd.monolayer_3band('MoS2') (3 coincident- position sublattices) gives a clean ~1.85 eV direct gap at K with no accidental degeneracy; the T-symmetric cubic Weyl semimetal (examples/weyl_lt.py, 3D, two sublattices, hoppings given by the model with same-\(\mathbf R\) forward/backward terms rather than relying on the automatic Hermitian-conjugate addition) reproduces its Weyl node exactly at \((\pi/2,\pi/2,\pi/2)\), matching the closed-form \(H(\mathbf k)=t[\cos k_x\,\sigma_x+\cos k_y\,\sigma_y+\cos k_z\,\sigma_z]\) to machine precision — confirming the automatic \(H_0+H_0^\dagger\) construction is safe even when a lattice's own hopping list already contains what looks like a manually-added return path. kite.repository.phosphorene.monolayer_4band() (4 sublattices with an out-of-plane buckling offset) gives a physically sane, anisotropic, direct ~1.5 eV gap at \(\Gamma\) — this case caught a real bug (a shape-mismatch crash whenever a sublattice position has more components than the lattice has primitive vectors), now fixed by zero-padding the in-plane cell-translation vector and truncating the dot product with \(\mathbf k\) to \(\mathbf k\)'s own dimensionality, exactly mirroring [plot_unit_cell][visualize-plot_unit_cell]'s top-down-projection convention for buckled lattices.

    function viz.compute_bands(lattice, k_path)

    Diagonalize [hamiltonian_k][visualize-hamiltonian_k] at every point of a k-path (raw points, or make_path(...)'s 3-tuple return value). Returns eigenvalues, shape (n_k, lattice.nsub), ascending order.
